1. Самое рискованное архитектурное решение в рамках задания это хранение данных в базе, в рамках прототипа такое допустимо но в продакшене лучше вынести файлы в объектное хранилище, а в бд хранить только путь к файлу, т.к. это напрямую влияет на производительность и быстрое заполнение бд
2. Сознательно не стал добавлять функционал администратора, т.к. на этапе прототипа роль пользователя можно менять напрямую через бд, при наличии времени добавил бы отдельные эндпоинты с расширенным функционалом и DataSeed для администратора.
Не реализовал аудит изменений статусов, в реальном сервисе бухгалтеру было бы полезно видеть историю изменений, а для пользователя можно было бы указывать причину отказа, решил бы созданием отдельной таблицы куда записывалось бы каждое изменение статуса.
Не добавил пагинацию, т.к. для небольшого количества пользователей на этапе разработки это не критично.
3. При высокой нагрузке необходимо:
вынести файлы в объектное хранилище
добавить пагинацию
добавить кеширование неизменяющихся данных
добавить SQRS и разнести команды и запросы по разным базам для более гибкого горизонтального масштабирования



Перед запуском приложения необходимо:

Запустить PostgreSQL

Указать SecretKey, длинной не менее 32 символов в appsettings
Указать строку подключения к базе данных в appsettings.
